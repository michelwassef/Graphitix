# Open Issues (Reviewed 2026-06-01)
# Ordered by descending priority. Closed and stale items were removed.

[open][P0][2026-06-01] Tab-scoped async safety is still fail-open: `workspaceTabs.isSessionMetaCurrent` returns `true` when session metadata is missing/invalid, so stale callbacks can bypass tab isolation guarantees. (from 2026-05-24 issue 55)

[open][P0][2026-06-01] Box frame/reserve authority is still not fully unified across flip/resize/reopen paths, causing tab-isolation-visible layout drift under switch/restore flows (e.g., `e2e/box.significance-layout.spec.js` currently fails at line 330 with large restored height drift). Consolidate to one authoritative frame snapshot -> reserve transfer -> draw contract. (merged from 2026-05-18 issue 28, 2026-05-21 issue 38, 2026-05-22 issue 42)

[open][P0][2026-06-01] Box reserve geometry regressions remain reproducible in unit tests: `__tests__/box.layoutReserves.regression.test.js` still fails 3 assertions (`yAxisSpan` drift on shrink with/without significance and `xAxisSpan` drift when significance toggles), confirming unresolved axis-stretch behavior beyond E2E symptoms. (new evidence 2026-06-01)

[open][P0][2026-06-01] Restore-draw scheduling remains fragmented and brittle across components: resize cache invalidation policy, reason-string suppression, and draw-option merge precedence are not centralized, so user-initiated redraws can still be suppressed or over-expanded after reopen. Replace with one shared scheduler/suppression + merge policy contract. (merged from 2026-05-31 issues 89, 90, 91, 95)

[open][P0][2026-06-01] `captureScatterRuntimeSnapshot` is still not a pure capture path; it writes through record->live rebind (`applyScatterOwnedRuntimeSlicesFromSnapshot`), so persist-time capture can mutate and clobber valid live scatter state. Split capture (live->record) from restore/bind (record->live). (from 2026-05-30 issue 82)

[open][P0][2026-06-01] Stats restore authority is still inconsistent between payload apply and activation/draw suppression flows; restored stats metadata can diverge from runtime signature inputs and be invalidated later. Standardize one payload-time stats hydration contract across components before context priming. (merged from 2026-05-22 issues 43, 44)

[open][P0][2026-06-01] Scatter/box payload serialization and control state acquisition still depend on live DOM in critical paths instead of state-first snapshots, keeping inactive-tab/headless flows fragile. Introduce a shared state-authoritative accessor contract. (from 2026-05-19 issue 32)

[open][P0][2026-06-01] Scatter restore still loses active data context after archive/recovery reopen: `e2e/stats.archive-reopen-tab-switch.persistence.spec.js` fails only for scatter in both Chromium and Firefox with `#scatterStatsStatus = "Statistics unavailable until data is loaded."` even after restored-tab activation. This is a true reopen-fidelity break (Rule 7) and likely a scatter hydration/activation ordering bug.

[open][P0][2026-06-01] Same-type scatter tab isolation is still violated under heavy archive restore in Firefox: `e2e/heavy.mixed-tabs.reopen-recovery.canvas.spec.js` restores both scatter tabs with first labels `["B1","B1"]` instead of preserving distinct `A*`/`B*` datasets, indicating cross-tab payload bleed during restore (Rule 6 violation).
