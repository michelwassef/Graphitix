[open][P1][2026-06-03] Bulk hot.loadData() does not sync the owner tab's payload, so a CSV/import on a single never-deactivated tab leaves tab.payload as the empty-default template while the live component (getPayload) holds the real data. ROOT CAUSE: Shared.hot only syncs tab.payload for per-cell afterChange (syncActiveTabPayloadDataChanges); afterLoadData just fires a draw and is deliberately classified non-user (so reopen stays invisible), so it neither syncs the payload nor marks the tab payloadDirty. Components then diverge: scatter ends up payloadDirty+null-payload, box happens to sync, but line/hist/heatmap end up clean+stale-empty. Any consumer that reads tab.payload via a lifecycle "skip if clean" flush therefore sees no data: writeRecoverySnapshot skipped/cleared the snapshot (no recovery until a tab switch flushed via getPayload), AND shouldWarnBeforeUnload suppressed the unsaved-changes warning. FIXED by forcing a live payload capture (captureLivePayload/allowSkipLivePayloadCapture:false) in documentState.flushActiveTabForRecovery and in sessionActions.shouldWarnBeforeUnload, so both read authoritative live state like a save does. Verified across all 5 table components in e2e/recovery.single-tab.live-capture.spec.js; unit guard in documentState.recoveryThrottle.test.js. Two follow-on defects this surfaced, both fixed: (a) FEEDBACK LOOP — forcing the full (non-skip) persist made persistActiveTabState.markSessionDirty fire on every recovery write (live-DOM layout drift => layoutChanged=true), bumping sessionRevision, which rescheduled another recovery write, whose capture re-detected drift... an unbounded heavy loop (endless debug logs, 500ms setTimeout violations). Fixed with a snapshotCapture intent flag that suppresses the markSessionDirty bump for recovery/beforeunload captures so they are revision-neutral (session.js); guard e2e/recovery.no-loop.spec.js. (b) WRITE LATENCY — the snapshot was written on a per-change debounce (RECOVERY_DELAY_MS, inflated to 5-8s for large payloads) that restarts on every change, so the post-import settle burst kept sliding it for several seconds; reloading in the first 1-2s lost recovery. Fixed with RECOVERY_MAX_WAIT_MS bounding the wait from the first pending change (~1s) regardless of churn; guard e2e/recovery.single-tab.e2e.spec.js. GENERALIZE (still open): the real homogeneity gap is that a user-initiated bulk load should sync tab.payload / mark dirty just like a cell edit — thread a user-vs-system source through loadData so afterLoadData can sync on user import without breaking reopen invisibility, then the "force live capture" guards become belt-and-suspenders. Audit remaining graphTabsHaveData/tabHasTableData callers for the same stale-clean-payload blind spot. Related to the 2026-05-19 "state-first snapshot accessor" P0 above.
